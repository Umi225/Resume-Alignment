# 编码规范 (Coding Rules)

> 最后更新：2026-05-13  
> 适用于：AI JD 对齐简历工作台

---

## 核心原则

### 1. 小步快跑，拒绝大规模重构
- **每次只完成一个小任务**，任务粒度控制在 1 个文件或 1 个组件
- 不允许一次性修改超过 5 个文件
- 不允许在实现 A 功能时"顺手"重构 B 模块
- 如需重构，必须先完成当前任务，再单独发起重构任务

### 2. 修改前先汇报影响范围
- 写代码前，先说明：
  - 本次修改涉及哪些文件
  - 是否影响已有功能
  - 是否有破坏性变更
- 得到确认后再开始修改

### 3. 不碰无关文件
- 只修改与当前任务直接相关的文件
- 不修改未在任务范围内的类型定义
- 不移动/重命名已有文件（除非任务明确包含）
- 不修改 `docs/` 目录下的文档（除非任务明确包含文档更新）

### 4. 保持类型一致
- 所有数据操作必须使用 `src/types/resume.ts` 中定义的类型
- 不随意扩展已有接口，如需扩展先讨论
- 不使用 `any`，如必须使用需注释说明原因

### 5. 组件设计约束
- 组件文件不超过 300 行，超过则拆分
- Props 必须显式定义接口，不使用隐式 `props`
- 业务组件放在 `components/` 下对应子目录，不随意平铺

---

## 代码风格

### TypeScript
- 使用 `interface` 定义对象类型，`type` 定义联合类型/工具类型
- 函数返回值显式声明（除非简单到一眼可判断）
- 使用可选链 (`?.`) 和空值合并 (`??`)，不用深层 `if` 嵌套

### React
- 使用函数组件 + Hooks
- 状态管理优先使用 Zustand，简单状态可用 `useState`
- 表单使用受控组件，复杂表单使用 `react-hook-form`

### Tailwind
- 使用工具类优先，复杂样式可提取为 `cn()` 工具函数
- 响应式前缀：`sm:`、`md:`、`lg:`
- 不使用 `@apply`（除非模板打印样式需要）

### 命名
- 组件：PascalCase（如 `ExperienceCard`）
- 函数/变量：camelCase（如 `getAssetFieldBySectionType`）
- 常量：SCREAMING_SNAKE_CASE（如 `DEFAULT_TEMPLATE_ID`）
- 文件：与默认导出同名（组件文件用 PascalCase，工具文件用 camelCase）

## 提交规范

- 提交前确认：只包含与当前任务相关的变更
- 提交信息格式：`类型: 简短描述`
  - 类型：`feat`（功能）、`fix`（修复）、`refactor`（重构）、`docs`（文档）、`types`（类型）
  - 示例：`feat: 添加经历卡片组件` / `fix: 修复日期解析错误`

---

## AI 协作规则

- 用户要求"先输出方案"时，不要直接写代码
- 用户要求"不要写页面"时，只输出类型/逻辑/文档
- 用户说"先让我审核"时，等待确认后再继续

# 开发规则

- 每完成一个功能后，必须进行 code review
- review 阶段禁止直接修改代码
- 必须分析：
  - 风险点
  - 状态冗余
  - 是否过度工程化
  - 是否影响现有功能
- 优先保持 MVP 简洁
- 非必要不新增抽象层
